CNC Basics

Hardware Structure

A simple structure looks like this:

  • Host computer
  • Link(s)
  • Drivers

The host processes a G-Code file, computes the tools trajectory in time and space, translates it into movement for the individual actuators (motors). The computations are governed by kinematics. The movement commands can be step/direction pulse or velocity/duration sequences. Drivers convert the movement commands into electrical signals for the motors. This could be a stepper motor driver with full/half/micro-stepping capabilities.

Ideally, soft- and hardware form a closed control loop for each motor. Since the control loop has a discrete nature (in contrast to closed analog control loops as in operational amplifier designs), actual values (position, speed, force, ...) must be sampled with precise timing (low jitter). Likewise, control output signals (steps, voltages, currents) must be set with precise timing. Failing to do so results in hard or non-tunable systems.

Computing trajectories requires floating-point math hardware (FPU), single-precision is acceptable for small machines, double or higher precision is ideal. Floating point math libraries are too slow to produce acceptable feed rates.

Computing trajecories and translation into timed control signal sequences doesn't have to be real-time, the path data must be produced just fast enough in order not to produes underrun conditions for the digital control loop. Therefore, this part of the software may run on an OS with relaxed realtime requirements (PC or SBC running Linux, 32-bit microcontroller with RTOS).

The digital control loop requires very hard realtime-capabilities (minimal jitter and latency). A Linux with realtime patches (LinuxCNC) is possible, but not ideal (acceptable data input and output jitter), a dedicated microcontroller is preferrable, digital logic (FPGA) or a microcontoller with specialized IO control modules (sequencers) is perfect for near-zero jitter and low latency.

Bit-banging with Microcontrollers

When you're looking for a microcontroller (MCU) for high-speed bit-banging and state machines, the bottleneck isn't usually just the clock speed—it's the latency between the CPU and the GPIO pins. In many modern MCUs, the I/O bus runs much slower than the core, which can "throttle" your bit-banging performance.

Suggestions:

Raspi 5 with BCM2712

This SoC is used in the Raspi 5. It has an external PIO module with 1 PIO block with 4 state machines, coupled to the BCM2712 CPU over a PCIe bus, which is not ideal for a CNC.

Teensy 4.1 i.MX RT1062

This SoC has no programmable IO blocks, but it's raw speed can provide accurate bit-bang signal generation up to 100 MHz.

RP2040

This Raspberry Pi model has 8 programmable input/output (PIO) blocks with FIFO buffers. They are running independently from the CPU can cannot be interrupted, and therefore produce pulse signals with the smallest possible jitter.

The PIO blocks are programmed with simple assembly-level instruction. The engine has loop and branch capabilities. The can be programmed for bit sequence outputs or high-speed quadrature encoder inputs.

With 8 PIO blocks, four closed loop controlled axes are possible.

RP2350

This Raspberry Pi model has 12 PIO-1 blocks and added DMA capabilities, which is sufficient for six closed loop controlled axes.

Clock frequency: 150 MHz, can be overclocked to 300 MHz.

Features "High-Speed I/O" ports that can toggle at the full speed of the AHB bus (up to 170 MHz).

Small in size. Clock frequency is 170 MHz.

Examples

RP2350 PIO Neopixel Driver

How the PIO State Machine Works: Think of the PIO as a middleman between your main C/C++ code and the GPIO pins. You push data into a FIFO (First-In-First-Out) buffer, and the PIO pulls it out and toggles pins based on your instructions.

.program ws2812
.side_set 1 opt

.define public T1 2	; This defines the interval durations
.define public T2 5
.define public T3 3

bitloop:
	out x, 1   side 0 [T3 - 1]		; Pull 1 bit from OSR, set pin LOW for T3 cycles
	jmp !x do_zero   side 1 [T1 - 1]		; If bit is 0, jump to do_zero, set pin HIGH for T1
do_one:
	jmp bitloop   side 1 [T2 - 1]		; Bit was 1: Keep pin HIGH for T2, then loop
do_zero:
	nop   side 0 [T2 - 1]		; Bit was 0: Set pin LOW for T2, then loop

The C code loads the PIO code:

To make this work, you need to set the PIO clock frequency to match the NeoPixel standard (usually 800kHz). Since our PIO program above uses 10 total cycles (T1+T2+T3=10), we set the PIO clock to 8MHz.

#include "pico/stdlib.h"
#include "hardware/pio.h"
#include "ws2812.pio.h"

void put_pixel(uint32_t pixel_grb) {
    // Push 32-bit GRB color data to the PIO FIFO
    pio_sm_put_blocking(pio0, 0, pixel_grb << 8u);
}

int main() {
    PIO pio = pio0;
    int sm = 0;
    uint offset = pio_add_program(pio, &ws2812_program);

    // Initialize with 8MHz frequency (10 cycles per bit at 800kHz)
    ws2812_program_init(pio, sm, offset, 2, 8000000, false);

    while (true) {
        put_pixel(0xFF0000); // Red (in GRB, this is Green=FF, Red=00...)
        sleep_ms(500);
        put_pixel(0x00FF00); // Green
        sleep_ms(500);
    }
}

Parallelism: You can run 8 separate NeoPixel strips on 8 different pins simultaneously using all 8 state machines, and the CPU performance wouldn't drop by a single percent.

On the newer RP2350, you can use the DMA (Direct Memory Access) to feed this PIO state machine. This allows you to animate thousands of LEDs by simply updating an array in RAM; the hardware handles the rest of the transmission in the background.

You said How would the planner calculate the data stream of a simple circle with a certain diameter in the x y plane for the PIO? What would the pattern look like? Gemini said To move a CNC machine in a circle, the Planner translates the abstract geometry of a circle into a series of discrete "steps" that the PIO can execute. Because a stepper motor can only move in fixed increments (e.g., 1.8 ∘ per step), a circle is actually a high-density polygon made of thousands of tiny X and Y movements. Here is how the data stream is calculated and what it looks like. 1. The Geometry: Bresenham or Sine/Cosine? In modern high-performance controllers like those on the RP2350, the Planner uses the vector approach: Sub-division: The circle is broken into very small linear segments (e.g., 0.1mm long). Trigonometry: For a circle with radius R, the position at angle θ is: X=R⋅cos(θ) Y=R⋅sin(θ) Step Conversion: The planner calculates the difference between the current position and the next (ΔX,ΔY) and converts that into the number of pulses required for the stepper drivers. 2. The Data Stream Structure The PIO needs a "heartbeat." To make a smooth circle, the Planner generates a Buffer in RAM. Each entry in the buffer tells the PIO two things: Direction/Step Bits: Which motors should move? Timer Value: How long should we wait before the next step? (This controls the speed). A simplified 32-bit PIO "Instruction" for the stream might look like this: | Bits 31–8 (24 bits) | Bits 7–4 (4 bits) | Bits 3–0 (4 bits) | | :--- | :--- | :--- | | Delay Count (Clock cycles) | Direction Pins (X,Y,Z,E) | Step Pins (X,Y,Z,E) | 3. The "Pattern" of the Circle If you visualized the pulses being sent to the X and Y axes during a 360° circle, the density of pulses follows a sine/cosine relationship: At 0° (East): The Y-axis is moving at maximum speed (dense pulses), while the X-axis is momentarily stationary (no pulses) as it changes direction. At 45° (North-East): Both X and Y axes are pulsing at an equal, steady rate (0.707× Max Speed). At 90° (North): The X-axis is at maximum speed, and the Y-axis is stationary. 4. How the RP2350 Executes It (DMA + PIO) This is where the "Ultra-fast" part comes in. You don't want the CPU to send every single pulse manually. The Buffer: Core 1 calculates 500 steps of the circle and stores them in an array in RAM. The DMA (The Postman): You configure the DMA controller to point at that array. The DMA is told: "Whenever the PIO's FIFO has room, grab the next 32-bit word from RAM and shove it in there." The PIO (The Performer): The PIO pulls the word, sets the pins, waits for the specified Delay Count, and then asks for the next word. The result: The CPU can go back to calculating the next part of the path (or handling a Web interface) while the hardware handles the high-speed pulse train with zero jitter. The Mathematical "Secret Sauce": Acceleration A simple circle isn't just about the path; it’s about Velocity. The planner must ensure the "Delay Count" starts high (slow), decreases (accelerates), stays steady, and then increases (decelerates) at the end of the arc. On an RP2350, the Cortex-M33 hardware float support allows you to calculate these S-curves in real-time without lagging the pulse stream. The Structure of a C2000 Project Unlike a single .ino file, a C2000 project is split into: The Hardware Abstraction Layer (HAL): Configures the PWMs, ADCs, and Timers. The Control Loop (ISR): A high-priority interrupt that runs at the frequency of your PWM (e.g., 20kHz). This is where the "State Machine" lives. The Background Task: Handles slower things like G-code parsing or temperature monitoring. 2. A "Hello World" Snippet (FOC Motor Control) This is what the heart of a TI C2000 motor control project looks like. It uses TI's InstaSPIN or Universal Lab libraries. C // --- The High-Speed Control Loop (Runs at 20kHz) --- __interrupt void mainISR(void) { // 1. Read the Currents and Voltages from the ADC HAL_readADCData(halHandle, &adcData); // 2.
